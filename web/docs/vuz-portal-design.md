# Кабинет вуза - проектный документ

Статус: ПРОЕКТ / не реализовано. Это дизайн-документ будущей фичи,
не описание текущего состояния приложения. После реализации каждой фазы
переносить факты в OVERVIEW/PAGES/API/DATABASE/FEATURES.

Родственный документ: `company-portal-design.md`. Кабинет вуза - это
облегчённый форк B2B-контура компаний: та же auth, тот же механизм
инвайтов и ролей, те же публичные страницы. Отличия помечены явно.
Go-to-market пакет для захода в вузы (письма, оффер, согласие, макет
отчёта) лежит в `vuz_package/` и связан с исследованием «Индекс джуна
2026» (`research_junior_market_design_v2.md` §5) - это первое касание,
кабинет - продукт удержания после первого «да».

## 1. Цель и рамки

Сейчас у платформы есть B2B-контур компаний (публикуют вакансии, разбирают
отклики) и кабинет кандидата. Вуза как сущности нет. Цель - дать карьерным
центрам вузов (ЦКС) свой контур:

- **брендированная витрина вуза** (публичная страница, замена мёртвого
  раздела «Трудоустройство» на сайте вуза, даёт .edu-бэклинк в нашу органику)
- **дашборд активности своих студентов** (агрегаты: сколько студентов вуза
  на платформе, что смотрят, куда откликаются, воронка)
- **канал связи со студентами** (рассылка подборок/анонсов от ЦКС)
- **выгрузки для отчётности** (мониторинг трудоустройства, отчёт ЦКС)

Позиционирование (не воевать с Факультетусом в лоб): Факультетус - учётная
система «для галочки», навязанная сверху; CareerLab - место, куда студенты
приходят сами (6000 органических визитов/мес). Кабинет вуза показывает, что
студенты делают на самом деле. Проговорить в питче явно: мы дополнение, не
замена обязательному мониторингу в Факультетусе.

Не входит в рамки (осознанно, на потом): интеграция с внутренними системами
вуза (LMS, 1С) - болото на месяцы; конструктор произвольных отчётов -
шаблонов и ручных выгрузок хватит; поимённая слежка за студентами - риск
потерять доверие аудитории (см. §7); ярмарки вакансий онлайн - территория
Факультетуса.

**Принципиальное упрощение против исходной концепции.** Верификация студента
= самодекларация: студент выбирает свой вуз из справочника при регистрации
или в настройках кабинета, и сразу засчитывается в агрегаты этого вуза.
Никакой корпоративной почты, кодов ЦКС и списков. Следствие: вуз видит
**только агрегаты, никогда не поимённо** - это снимает основной блок по
152-ФЗ и делает MVP реалистичным. Опциональная «подтверждённая» доля
студентов (по домену почты) - потом, если вуз попросит цифру построже.

## 2. Персоны и сценарии

### Персоны

- **Руководитель ЦКС** - показывает дашборд проректору, отвечает за KPI
  охвата, шлёт рассылки, выгружает отчёты (owner кабинета вуза)
- **Сотрудник ЦКС** - операционка: рассылки, наполнение витрины (staff)
- **Студент/выпускник** (существующая персона кандидата) - выбирает свой вуз
  в профиле, дальше пользуется платформой как обычно
- **Владелец платформы** - заводит вуз и приглашает ЦКС, модерирует витрину,
  делает concierge-выгрузки первые месяцы

Важно: в отличие от компаний, вуз **не создаёт сам себя** и не публикует
вакансии - значит нет анти-скам вектора и премодерации контента вуза.
Онбординг вуза ручной, через владельца платформы (продажа пилота).

### Сценарий A. Заход вуза (ручной онбординг)

1. Владелец платформы продал пилот (через `vuz_package/`: письмо в ЦКС +
   оффер + макет отчёта). Получено «да».
2. Владелец в `/admin` создаёт запись вуза: name, город, регион, logo_url,
   slug. Статус сразу `active` (нет самозаписи - нет и премодерации вуза).
3. Владелец приглашает руководителя ЦКС по email (механика инвайта как у
   компаний: token_hash + expires_at, 7 дней).
4. ЦКС принимает приглашение -> становится owner кабинета вуза.
5. Владелец отдаёт ЦКС тексты рассылки студентам (`vuz_package/03_*`) со
   ссылкой с UTM-кодом вуза -> студенты приходят и выбирают вуз в профиле.

Ветвление: вуз уже заведён (был в справочнике для выбора студентами, но без
ЦКС) -> просто привязать owner-инвайт к существующей записи, агрегаты по
самодекларации уже могли копиться.

### Сценарий B. Студент выбирает вуз (ядро атрибуции)

1. **При регистрации**: опциональное поле «Ваш вуз» (селект по справочнику
   `universities`, поиск по названию). Можно пропустить.
2. **В кабинете** `/office/settings`: блок «Учёба» - вуз, факультет
   (текст, опц.), курс (опц.), год выпуска (опц.). Редактируется всегда.
3. Как только `university_id` проставлен - студент попадает в агрегаты вуза
   (счётчик студентов, воронка, топ-интересы). Ретроактивно: его прошлые
   сохранёнки/отклики уже учитываются в срезах.
4. Смена вуза / очистка поля - студент выпадает из агрегатов старого.

Мост к go-to-market: если студент пришёл по ссылке с UTM-кодом вуза
(`?utm_source=vuz&utm_campaign=<slug>`), предзаполнять селект вуза этим
значением (мягко, с возможностью изменить) - это связывает исследовательский
трафик с атрибуцией без принуждения.

### Сценарий C. Команда ЦКС и роли

1. Owner в разделе «Команда» вводит email коллеги и роль.
2. Письмо-приглашение с токеном (тот же механизм, что `company_invites`).
3. Коллега переходит: нет аккаунта - регистрируется, есть - логинится;
   членство активируется.
4. Owner может менять роли и деактивировать участников.
5. Один аккаунт может состоять в нескольких вузах и/или компаниях
   (переключатель активной сущности, cookie как `cl_active_company`).

Матрица ролей (двух хватает для ЦКС):

| Действие | owner | staff |
|----------|-------|-------|
| Дашборд, выгрузки | + | + |
| Витрина вуза (ред.) | + | + |
| Рассылка студентам | + | + |
| Команда: приглашать, менять роли | + | - |
| Передать владение | + | - |

### Сценарий D. Дашборд (главный экран, что показывают проректору)

Одна страница агрегатов по студентам этого вуза (все цифры - только по тем,
кто сам выбрал вуз):

- Студентов вуза на платформе + динамика за месяц/семестр.
- Активность: просмотры вакансий, отклики, анализы резюме за период.
- Воронка: выбрал вуз -> заполнил профиль/загрузил резюме -> откликнулся ->
  приглашение -> (оффер - если научимся отмечать, см. §9).
- Топ-направления интереса (какие сферы/компании смотрят и куда откликаются).
- Сравнение с анонимным бенчмарком по всем вузам («ваши откликаются на N%
  чаще среднего»).
- Сегмент «неактивные» - **только число** (сколько выбравших вуз не заходили
  N дней), без поимённого списка (см. §7).

Cold start (важно, проговорить в питче честно): на старте пилота выбравших
вуз студентов мало, дашборд наполняется по мере рассылки. Поэтому
операционный порядок обратный питчу: витрина + рассылка = движок
привлечения, дашборд = отложенная выплата. В демо показываем мокап с
модельными числами (`/vuz-demo`), в живом кабинете - плашку «данные
накапливаются» пока студентов меньше порога.

### Сценарий E. Витрина вуза (публичная страница, SEO)

1. ЦКС в `/vuz/showcase` заполняет: описание карьерного центра, контакты,
   анонсы мероприятий, опционально публичные цифры.
2. Публичная страница `/universities/[slug]` (форк `/companies/[slug]`):
   лого вуза, блок ЦКС, анонсы, вакансии/стажировки от партнёров (позже),
   публичные цифры для абитуриентов.
3. Вуз ставит ссылку с сайта (`sitename.ru/career` -> витрина) - .edu-уровня
   бэклинк, золото для нашей органики; заодно замена мёртвого раздела.
4. Публичные цифры показываются только выше порога (не позорить пустотой) и
   только агрегированно; тумблер «показывать/скрыть» у ЦКС.

### Сценарий F. Рассылка студентам (канал связи)

1. ЦКС в `/vuz/broadcast` собирает сообщение: заголовок, текст, опц.
   подборка вакансий/статей, ссылка на анонс/ярмарку.
2. Аудитория - только студенты, выбравшие этот вуз и не отписавшиеся.
3. Каналы MVP: in-app (колокольчик, таблица `notifications`) + email
   (через `notification_prefs`, у студента тумблер «сообщения от вуза»).
   Позже - Telegram-бот (у CareerLab уже есть TG-аудитория).
4. Анти-спам: лимит частоты (напр. не чаще 1 рассылки в неделю на вуз),
   обязательная отписка, премодерация первых рассылок владельцем платформы.
5. Статистика доставки/открытий - сама по себе отчётная метрика для ЦКС.

Приоритет ниже дашборда/витрины: больше всего инженерии и рисков со спамом;
в питче показать мокапом, строить после первого «да».

## 3. Модель данных

Все новые таблицы - в пользовательском Postgres (как auth, кабинет,
компании), с миграциями. Максимально переиспользуем существующее:
`careerlab_accounts` (auth), `notifications` / `notification_prefs`
(уведомления), механику `company_invites` (инвайты).

```sql
-- Вузы (справочник + владеющие ЦКС). Заводит владелец платформы, не самозапись.
CREATE TABLE universities (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  slug          text UNIQUE NOT NULL,
  name          text NOT NULL,
  short_name    text,                 -- «МИСИС», для селекта и бейджей
  city          text,
  region        text,
  logo_url      text,
  description    text,                -- блок ЦКС на витрине
  contacts      jsonb NOT NULL DEFAULT '{}',  -- email/тел/адрес ЦКС
  status        text NOT NULL DEFAULT 'active', -- active | hidden
  public_stats  boolean NOT NULL DEFAULT false, -- показывать цифры на витрине
  email_domains text[],               -- для опц. «подтверждённой» доли (§9)
  created_by    bigint REFERENCES careerlab_accounts(id),
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now()
);

-- Членство сотрудников ЦКС (роль аккаунта в вузе)
CREATE TABLE university_members (
  university_id uuid   REFERENCES universities(id) ON DELETE CASCADE,
  account_id    bigint REFERENCES careerlab_accounts(id) ON DELETE CASCADE,
  role          text   NOT NULL,   -- owner | staff
  status        text   NOT NULL DEFAULT 'active',  -- active | disabled
  invited_by    bigint,
  created_at    timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (university_id, account_id)
);

-- Приглашения в ЦКС (механика как company_invites: token_hash + expires_at)
CREATE TABLE university_invites (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  university_id uuid REFERENCES universities(id) ON DELETE CASCADE,
  email         text NOT NULL,
  role          text NOT NULL,
  token_hash    text NOT NULL,
  invited_by    bigint,
  expires_at    timestamptz NOT NULL,
  accepted_at   timestamptz,
  UNIQUE (university_id, email)
);

-- Учебный профиль студента (самодекларация; отдельно, чтобы не пухли accounts)
CREATE TABLE student_profiles (
  account_id      bigint PRIMARY KEY REFERENCES careerlab_accounts(id) ON DELETE CASCADE,
  university_id   uuid REFERENCES universities(id) ON DELETE SET NULL,
  faculty         text,           -- опц., самодекларация
  study_year      smallint,       -- опц., курс 1..6
  graduation_year smallint,       -- опц.
  email_verified_domain boolean NOT NULL DEFAULT false, -- §9, пока всегда false
  updated_at      timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX ON student_profiles (university_id);
```

Дашборд и срезы считаются джойном `student_profiles` (по `university_id`)
с уже существующими таблицами активности: `user_saved_vacancies`,
`applications`, `vacancy_stats` (агрегаты), история анализа резюме.
Отдельных «событийных» таблиц под вуз не заводим в MVP - агрегируем то, что
уже пишется. Тяжёлые агрегаты - материализованная вьюха или ночной пересчёт
в `university_daily_stats(university_id, date, metric, value)`, если запросы
станут дорогими (не раньше реальных объёмов).

Рассылки переиспользуют `notifications` (type `vuz_broadcast`, payload с
university_id и телом) и email через `notification_prefs` (новый ключ
`vuz_broadcast`). Отдельная таблица `university_broadcasts(id, university_id,
title, body, payload, audience_size, sent_at, sent_by)` - для истории и
статистики доставки.

## 4. API (эскиз)

Роуты вуза требуют session-cookie + членства в вузе с ролью. Хелпер
`lib/auth/university-guard.ts`: `requireUniversityRole(universityId, minRole)`
(зеркало `company-guard.ts`).

```
Справочник (публичный, для селекта студента)
GET    /api/universities?q=            поиск вуза по названию (для регистрации/настроек)

Профиль студента (самодекларация)
GET    /api/office/student-profile      мой учебный профиль (auth)
PUT    /api/office/student-profile      выбрать/сменить вуз, факультет, курс (auth)

Кабинет вуза
GET    /api/universities/mine           мои вузы + роли (auth)
GET    /api/university/[id]/dashboard    агрегаты дашборда (member)
GET    /api/university/[id]              карточка/настройки витрины (member)
PATCH  /api/university/[id]              ред. витрины, контактов, public_stats (member)

Команда ЦКС
GET    /api/university/[id]/members      список (member)
POST   /api/university/[id]/invites      пригласить (owner)
POST   /api/university-invites/accept    принять по токену
PATCH  /api/university/[id]/members/[acc] роль/деактивация (owner)

Рассылка
GET    /api/university/[id]/broadcasts   история + статистика (member)
POST   /api/university/[id]/broadcasts   создать/отправить (member, премодерация)

Выгрузки (concierge-MVP: сначала владелец платформы руками)
GET    /api/university/[id]/export?type=monitoring|semester  CSV/XLSX (member)

Админ (владелец платформы)
POST   /api/admin/universities           создать вуз + пригласить owner ЦКС
GET    /api/admin/universities           список вузов, статусы
POST   /api/admin/broadcasts/[id]/review approve/reject первой рассылки вуза
```

Роль владельца платформы - существующий флаг `is_platform_admin` / whitelist
email (как у модерации компаний).

## 6. Приватность и 152-ФЗ

Ключ, снимающий большинство рисков: **самодекларация + только агрегаты**.

- Вуз никогда не видит студентов поимённо, переписку, отдельные отклики.
  Все экраны кабинета вуза - агрегаты. Сегмент «неактивные» - число, не список.
- Порог показа: метрика не отображается (и в кабинете, и на витрине), пока
  в группе меньше K студентов (напр. K=5) - защита от деанонимизации малых
  выборок.
- Отдельного согласия под вуз не требуется: студент добровольно указывает
  вуз, данные обезличены в агрегате. В `/consent` и `/privacy-policy` - абзац,
  что указанный вуз используется для обезличенной статистики карьерного
  центра и для рассылок ЦКС (с правом отписки).
- Рассылки ЦКС - через тумблер `vuz_broadcast` в `notification_prefs` и
  обязательную отписку в каждом письме.
- «Подтверждённая» доля по домену почты (§9) - тоже только счётчик, не
  раскрывает личности.

## 7. Безопасность и качество

- **Достоверность самодекларации**: студент может указать чужой вуз. Для
  вэнити-агрегатов MVP приемлемо (низкие ставки). Митигация от накрутки:
  один вуз на аккаунт, троттлинг смены вуза, флаг подтверждённой доли для
  вузов, которым нужна цифра построже.
- **Анти-спам рассылок**: лимит частоты на вуз, премодерация первой рассылки,
  обязательная отписка, статистика жалоб.
- **Витрина**: контент ЦКС проходит лёгкую модерацию владельцем платформы
  (репутация выдачи), публичные цифры - только выше порога K.
- **Cold-start честность**: не показывать вузу/абитуриентам пустые или
  мизерные цифры (плашка «накапливаем данные»).
- **SEO-гигиена**: `/universities/[slug]` - канонические, в sitemap только
  `active`+заполненные; `hidden` и пустые исключены.

## 8. Фазы

**Фаза 1 - витрина + атрибуция (самое дешёвое и с максимальным рычагом):**
справочник `universities` (сидинг реестра вузов), выбор вуза студентом
(регистрация + `/office/settings`, таблица `student_profiles`), публичная
витрина `/universities/[slug]`, лендинг `/for-universities`, демо `/vuz-demo`
для питча, админ-онбординг вуза + инвайт owner ЦКС. Даёт бэклинк и начинает
копить атрибуцию сразу, ещё до дашборда.

**Фаза 2 - кабинет ЦКС:** членство/роли (owner/staff через инвайты, форк
company-контура), дашборд агрегатов, срезы по студентам, concierge-выгрузки
(CSV руками владельцем платформы). Порог K и плашки cold-start.

**Фаза 3 - удержание:** рассылка студентам (in-app + email), история и
статистика доставки, автоотчёт раз в месяц на почту ЦКС; переключатель
активной сущности вуз/компания. Позже: TG-бот, партнёрские вакансии на
витрине, мост к B2B («компания X интересуется студентами вуза Y»),
подтверждённая доля по домену почты.

**Для питча (сейчас, 0 разработки):** 4 HTML-мокапа в дизайн-системе
(Unbounded + лайм) - дашборд, витрина, список студентов, рассылка;
переиспользовать паттерн `/office-demo`.

## 9. Открытые вопросы (решить до реализации)

1. **Сидинг справочника вузов**: залить готовый реестр (топ РФ-вузов) или
   заводить по мере пилотов? Для быстрого старта - небольшой сид (100-200
   вузов) + добор владельцем платформы.
2. **Подтверждённая доля по домену почты**: нужна ли вузу-пилоту цифра
   «N из них с корпоративной почтой»? Схема готова (`email_domains`,
   `email_verified_domain`), включаем только если попросят.
3. **Порог K** для показа агрегатов (5? 10?) - согласовать с первым вузом.
4. **Отметка «оффер»** в воронке: сейчас платформа не знает про офферы
   (отклики только до invited). Либо самоотметка студентом «получил оффер»,
   либо оставить воронку до invited. Влияет на главную метрику дашборда.
5. **МИСИС как пилот**: «бесплатный кабинет в обмен на рассылку и лого» или
   сразу партнёрство? (влияет на приоритет фаз - при обмене на рассылку
   Фаза 3 нужнее раньше).
