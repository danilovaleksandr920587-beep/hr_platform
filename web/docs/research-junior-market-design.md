# Исследование рынка джунов - дизайн-документ

Статус: ПРОЕКТ / не реализовано. Дизайн спецпроекта "исследование рынка
джуниор-вакансий": цели, данные, методология, упаковка, дистрибуция.
Рабочее название отчёта: "Сколько стоит 'без опыта': рынок джунов 2026".
После запуска переносить факты в OVERVIEW/PAGES/FEATURES.

## 1. Зачем это исследование (стратегическая рамка)

Исследование - первый шаг партнёрской стратегии Career Lab, а не разовый
пиар. Оно решает четыре задачи одновременно:

1. **Медийный вес.** Публичный отчёт с "неудобными цифрами" о рынке джунов
   даёт повод писать о платформе СМИ и ТГ-каналам. Career Lab становится
   источником, на который ссылаются.
2. **Вход к вузам и студоргам.** Опрос (слой 3 данных) распространяется
   через центры карьеры и студенческие организации. Просьба "помогите
   собрать анкеты, ваш вуз попадёт в отчёт" - лёгкий первый контакт,
   который позже конвертируется в события и Лигу вузов.
3. **Тёплые контакты с компаниями.** 2-3 компаниям предлагаем дать
   экспертный комментарий в отчёт (бесплатно) - первое касание для
   будущих продаж (размещение, спецпроекты, кейс-чемпионаты).
4. **SEO и трафик.** Интерактивный отчёт на lab-career.ru собирает
   ссылочную массу и органический трафик; внутри отчёта зашиты ссылки
   на анализатор резюме и калькулятор зарплат (конверсия в регистрации).

Релиз исследования - трамплин для следующего спецпроекта ("Неделя найма
без опыта"): сначала показываем проблему цифрами, затем делаем событие,
которое её решает.

## 2. Исследовательские вопросы и целевые заголовки

Отчёт держится на 4 "гвоздях" - цифрах, ради которых его будут
пересылать. Каждому соответствует исследовательский вопрос:

| # | Вопрос | Целевой заголовок (шаблон) | Источник данных |
|---|--------|----------------------------|-----------------|
| 1 | Какая доля вакансий с пометкой junior/стажёр требует опыт 1-3 года? | "N% 'джуновских' вакансий требуют опыт" (мем про джуна с 3 годами опыта, оцифрован) | HH API + парсер |
| 2 | Какая доля джун-вакансий публикуется без зарплатной вилки? Каковы медианы там, где вилка есть? | "M% вакансий для новичков скрывают зарплату"; "медиана джуна в X - Y руб." | HH API + парсер |
| 3 | Сколько откликов и месяцев нужно джуну до первого оффера? Какая доля откликов остаётся без ответа? | "K откликов и L месяцев до первой работы"; "P% откликов джунов игнорируются" | Опрос |
| 4 | В каких сферах вход реально открыт (доля вакансий exp=none), а где "джуниор" - фикция? | "Куда джуна возьмут на самом деле: рейтинг сфер по открытости входа" | HH API + парсер |

Приоритет: вопрос 3 - самый ценный, потому что сторону кандидата
в связке с рынком вакансий не покажет ни hh, ни Хабр Карьера.
Вопросы 1-2 - самые виральные. Вопрос 4 - "полезность", ради которой
отчёт сохраняют.

Дополнительные разрезы (если данных хватит): ожидания студентов
по зарплате (опрос) vs реальные вилки (вакансии); удалёнка vs офис
для джунов; стажировки - оплачиваемые vs бесплатные (доля).

Чего НЕ делаем в первом выпуске:
- Рейтинг конкретных компаний поимённо (отзывчивость, зарплаты) -
  юридически и репутационно тонко, данных по откликам через платформу
  пока мало. Это "Индекс отзывчивости" - второй сезон, когда накопятся
  статусы откликов в `applications`.
- Прогнозы рынка - не наша экспертиза, легко облажаться.

## 3. Данные: три слоя

### Слой 1. Собственный парсер (junior_jobs_parser)

Что есть: ~сотни вакансий в проде, поля title, company, description
(блоки задачи/требования/условия), sphere, exp (none/lt1/1-3/gte3),
format, employment_type, salary_min/max, bonus_tags, источник.
Источники: ТГ-каналы + карьерные страницы компаний (адаптеры VK,
Kaspersky, T-Bank, JetBrains).

Роль в исследовании: не основа выборки (мало), а **дифференциатор** -
срез "скрытого рынка" вне hh: ТГ-каналы и прямые карьерные страницы.
Отдельная врезка в отчёте: чем вакансии вне агрегаторов отличаются
(зарплаты, требования, доля стажировок).

### Слой 2. Открытый API hh.ru (основной массив)

Публичный API (api.hh.ru/vacancies) отдаёт вакансии с полями experience,
salary, area, professional_roles, schedule, published_at. Целевой объём:
10-30 тыс. вакансий уровня junior за фиксированный период (например,
2 календарных месяца сбора).

Стратегия сбора:
- поисковые запросы по маркерам: junior, джуниор, стажёр, стажировка,
  trainee, intern, "без опыта", "начинающий" + фильтр experience
  (noExperience, between1And3) по целевым проф. ролям (IT, дизайн,
  маркетинг, аналитика - те же sphere, что на платформе)
- дедупликация по id; снятие снапшотов раз в N дней (API отдаёт только
  активные вакансии, историю надо копить самим)
- сохранять сырые JSON (воспроизводимость методологии)

Ограничения API учесть в коде: пагинация максимум 2000 вакансий на
запрос (дробить по датам/ролям/регионам), rate limit, обязательный
User-Agent. Использование - в рамках условий API hh (некоммерческое
исследовательское использование, без перепубликации полных текстов
вакансий; в отчёте - только агрегаты).

Классификация "джуновости" - переиспользовать логику
`adapters/entry_level.py` (_POSITIVE_RE/_NEGATIVE_RE + detect_exp):
она уже отделяет джун-вакансии от middle/senior по тексту. Это же -
основа для метрики вопроса 1: вакансия *называется* джуновской
(позитивный маркер в title), но *требует* опыт (experience поле hh
или detect_exp по описанию = 1-3/gte3).

### Слой 3. Опрос джунов и студентов

Единственный источник стороны кандидата. Цели по выборке:
- минимум 500 полных ответов (порог публикации), цель 1000+
- костяк: студенты старших курсов и выпускники 0-2 лет, ищущие/искавшие
  первую работу в IT/дизайне/маркетинге/аналитике

Блоки анкеты (~5-7 минут, до 20 вопросов):
1. Скрининг: статус (студент/выпускник/работаю), направление, город
2. Поиск: ищу/искал/нашёл; длительность поиска (мес.); число откликов
   всего; число ответов любых; число приглашений; число офферов
3. Игнор: доля откликов без какого-либо ответа (оценочно, шкалой)
4. Зарплата: ожидаемая на первой работе; полученная (если нашёл)
5. Каналы: где ищет (hh, ТГ-каналы, знакомые, карьерный центр вуза...)
6. Опционально: вуз (свободное поле + подсказки) - для разрезов
   по вузам-партнёрам и для отчётов вузам
7. Контакт (опционально): email для получения отчёта = магнит
   подписки/регистрации

Дистрибуция анкеты - и есть партнёрская механика:
- студорги и ТГ-сообщества: прямые договорённости, взаимный пиар
- центры карьеры вузов: письмо-питч "ваш вуз в исследовании + вы
  получите срез по своим студентам первыми"
- собственные каналы: сайт (баннер), база email, соцсети
- у каждого канала - UTM/код источника, чтобы считать вклад партнёров
  и благодарить поимённо в отчёте

Инструмент: Яндекс Формы или своя форма на платформе (плюс своей -
магнит регистраций и полный контроль данных; минус - разработка).
Для MVP допустимы Яндекс Формы, выгрузка в CSV для анализа.

ПД: анкета анонимна; email - отдельным опциональным полем с явным
согласием на рассылку (152-ФЗ); сырые данные не публикуются,
только агрегаты; в вузовских срезах - минимум 30 ответов на вуз,
иначе не показываем (защита от деанонимизации).

## 4. Методология: определения и метрики

Публикуемый раздел "Методология" обязателен - это щит от критики
и признак серьёзности. Ключевые определения:

**Джун-вакансия** (для слоёв 1-2): вакансия, у которой выполняется
хотя бы одно: (а) позитивный маркер в заголовке (junior/джуниор/стажёр/
trainee/intern/начинающий/без опыта - regex из entry_level.py);
(б) experience = noExperience в HH API; (в) detect_exp по тексту =
none/lt1. Негативные маркеры (senior/lead/ведущий/...) исключают.

**"Называется джуновской, но требует опыт"** (метрика вопроса 1):
позитивный маркер в title И (hh experience = between1And3 или строже,
ИЛИ detect_exp по требованиям = 1-3/gte3). Считаем долю от всех
вакансий с позитивным маркером в title.

**Зарплатная непрозрачность** (вопрос 2): доля вакансий с salary = null.
Медианы считаем по середине вилки (или единственной границе), gross/net
нормализуем по флагу hh; свои данные - salary_min/max как есть, с
оговоркой. Медианы публикуем только для ячеек с n >= 50.

**Открытость входа** (вопрос 4): по сферам - доля вакансий
с experience = noExperience среди всех джун-вакансий сферы + абсолютное
число. Ранжируем сферы по комбинации доли и объёма.

**Метрики опроса** (вопрос 3): медианы (не средние - выбросы), число
откликов до первого оффера, месяцы поиска, доля безответных откликов.
Разрезы: направление, город (Мск/СПб/другие), статус.

Правила честности:
- каждая цифра в отчёте сопровождается базой: "среди N тыс. вакансий,
  опубликованных на hh.ru и в профильных ТГ-каналах в [период]"
- не экстраполируем на "весь рынок России" - только на описанную выборку
- выборка опроса - самоотбор через партнёрские каналы; прямо пишем это
  ограничение в методологии
- сырые агрегаты и определения воспроизводимы: публикуем список
  regex-маркеров и период сбора

## 5. Артефакты релиза

| Артефакт | Носитель | Назначение |
|----------|----------|------------|
| Интерактивный отчёт | страница на lab-career.ru (каркас - существующий /research с Chart.js) | главный носитель, цель всех ссылок, SEO |
| PDF-версия | файл 15-25 стр. | для СМИ, вузов, компаний; "официальный документ" |
| Пресс-кит | архив/страница: 5 цифр-цитат, графики PNG, комментарий основателя, boilerplate о платформе | журналисту нечего делать, кроме копипаста |
| Карточки для ТГ | 4-6 изображений (по одному "гвоздю") | посев по ТГ-каналам, репосты |
| Врезки-CTA внутри отчёта | раздел про отклики -> анализатор резюме; про зарплаты -> калькулятор | конверсия читателя в регистрацию |
| Персональные срезы вузам | mini-PDF на вуз (при n >= 30) | благодарность партнёрам + повод для следующего контакта |

Технические работы в репо (по фазам, отдельными задачами):
1. Сборщик HH API: `junior_jobs_parser/hh_research_collector.py` -
   отдельный скрипт, не смешивать с продовым пайплайном вакансий;
   складывает сырые JSON в каталог данных исследования
2. Анализ: скрипт агрегации (pandas или чистый python) -> JSON с
   готовыми рядами для графиков
3. Страница отчёта: новый роут (например `/research/junior-market-2026`),
   компонент по образцу ResearchSalaryDashboard, данные - статический
   JSON из шага 2 (не живая БД: отчёт - зафиксированный снапшот)
4. Опционально: своя форма опроса + таблица ответов (если не Яндекс Формы)

## 6. Дистрибуция релиза

Последовательность (неделя релиза):
1. **Эксклюзив одному СМИ** (vc.ru / РБК Трендс / Forbes Education -
   выбрать одно) за 2-3 дня до публичного релиза: им - первые цифры,
   нам - большая площадка
2. **Публикация лендинга** + PDF в день X
3. **Веерный посев**: ТГ-каналы про IT-карьеру и поиск работы (в т.ч.
   каналы-источники парсера - с ними уже есть точки контакта), HR-каналы,
   каналы студоргов-партнёров; каждому крупному каналу - "своя" цифра
4. **Рассылка вузам-партнёрам** с персональными срезами
5. **Хабр-статья** (перепаковка отчёта в лонгрид) через 3-7 дней -
   вторая волна
6. **Компании**: письмо "мы выпустили исследование о джун-рынке,
   у нас аудитория X - вот что мы можем для вашего найма" - начало
   продажных разговоров

## 7. Таймлайн (~8 недель)

| Недели | Работа |
|--------|--------|
| 1-2 | Сборщик HH API + первый прогон; финализация методологии; анкета опроса; питч-письма вузам/студоргам |
| 2-6 | Полевой сбор: анкета в каналах партнёров; накопление снапшотов вакансий; договорённость со СМИ об эксклюзиве |
| 5-7 | Анализ, тексты отчёта, лендинг, PDF, пресс-кит, карточки |
| 8 | Релиз по плану раздела 6 |

Критический путь - опрос: если к неделе 5 меньше 500 ответов,
сдвигаем релиз, а не публикуем слабые данные (порог - жёсткий).

## 8. Метрики успеха спецпроекта

- >= 500 (цель 1000) ответов опроса
- >= 10 тыс. вакансий в выборке HH
- >= 5 публикаций в СМИ/крупных каналах со ссылкой на отчёт
- >= N новых регистраций на платформе за месяц релиза (базлайн снять
  до релиза; следить через UTM)
- >= 5 вузов/студоргов, с которыми появился рабочий контакт
- >= 3 компании, с которыми начат продажный диалог после рассылки
- ссылочная масса: новые referring domains на lab-career.ru (Ahrefs)

## 9. Риски

| Риск | Митигация |
|------|-----------|
| Опрос не набирает 500 ответов | начать дистрибуцию с недели 2, не 4; резерв - платный посев в 1-2 ТГ-каналах; жёсткий порог публикации |
| Претензии к методологии | публичный раздел "Методология", честные рамки выборки, никаких экстраполяций |
| Условия использования HH API | только агрегаты в публикации, без перепубликации текстов вакансий; сырые данные не раздаём |
| Цифры окажутся "скучными" (мем не подтвердится) | это тоже заголовок ("на самом деле опыт требуют только N%"); гвозди 2-4 независимы от гвоздя 1 |
| ПД в опросе | анонимность, опциональный email с согласием, порог n >= 30 на вузовский срез |
| Один человек не тянет объём | резать скоуп в пользу слоёв 2-3 (HH + опрос); слой 1 (свой парсер) - опциональная врезка |

## 10. Открытые вопросы (решить до старта)

1. Период сбора вакансий: 1 или 2 месяца? (2 месяца надёжнее, но
   сдвигает релиз; можно собирать параллельно с опросом - тогда 2)
2. Опрос: Яндекс Формы (быстро) или своя форма (магнит регистраций)?
   Рекомендация для MVP: Яндекс Формы, свою - к второму сезону
3. Какому СМИ предлагать эксклюзив первым?
4. Брендинг отчёта: делать ли постоянное имя серии (например,
   "Индекс джуна" / "Junior Market Report"), чтобы второй сезон
   узнавали? Рекомендация: да, серийность = капитализация
5. Бюджет на дизайн PDF/карточек и (резерв) платный посев анкеты
