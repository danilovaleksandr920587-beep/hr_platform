# Фичи по модулям

## Вакансии
- Каталог с фильтрами: сфера, опыт, формат работы, тип занятости, зарплата
- Полнотекстовый поиск на русском (tsvector в Postgres)
- Карточка вакансии: `components/VacancyCard.tsx`, страница - SSG
- Сохранение в кабинет (сердечко), избранные вакансии на главной
- Наполнение: python-парсер `junior_jobs_parser/` + ручная модерация

## База знаний (SEO-ядро сайта)
- Статьи по 6 кластерам: резюме, собеседование, тестовые, зарплата, отклики, карьера
- Хаб-страницы категорий с приоритетами (`lib/knowledge-clusters.ts`)
- Фильтры по категории и уровню (Новичок/Продвинутый)
- Пайплайн публикации: `article_pipeline.py` (соседний проект /Users/user/claude,
  правила SEO - в его CLAUDE.md)
- `llms.txt` со списком всех статей для LLM-краулеров

## Инструменты
- **Анализатор резюме**: загрузка PDF/DOCX или вставка текста -> парсинг ->
  YandexGPT -> балл + чек-лист рекомендаций. Для залогиненных история
  сохраняется в кабинет
- **Калькулятор зарплат**: 13 направлений x 4 уровня x 5 локаций. Данные в
  `lib/data/salary.ts` (единый источник и для карточки в /office): базовая
  медиана по Москве x коэффициент города x индексация годовым трендом от
  даты калибровки `SALARY_AS_OF`. Обновление = поправить базы и дату

## Кабинет компании (B2B, фаза 1)
- Регистрация компании (любой залогиненный аккаунт), верификация владельцем
  платформы через `/admin/moderation` (анти-скам: аудитория - джуны)
- Команда: owner приглашает по email-токену (7 дней), роли owner/recruiter,
  деактивация участников, защита от удаления последнего owner
- Вакансии компании: CRUD в той же таблице `vacancies` (source=company),
  статусы draft -> pending_review -> published/rejected -> archived;
  premoderation, для trusted-компаний - автопубликация. Форма описания -
  структурная (Задачи/Требования/Условия + О команде): пишется в
  `description_blocks` + склейка в `description` для поиска/SEO; старые
  плоские описания рендерятся через фолбэк «О роли»
- Логотип компании: поле «Логотип (URL)» в создании и настройках (только
  https-ссылка); дублируется в `vacancies.company_logo_url` при создании
  и правке вакансии, при смене в настройках синкается во все вакансии
  компании (`syncCompanyBrandingToVacancies`)
- Отклики: apply_mode=internal - форма на платформе (резюме PDF/DOCX,
  сопроводительное, контакт), у компании воронка со статусами
  new/viewed/invited/rejected, у кандидата - `/office/applications` с отзывом.
  В карточке отклика - название вакансии (join по slug на уровне страницы)
- Email-уведомления: новый отклик (команде), смена статуса (кандидату),
  приглашение, итоги модерации, алерты админам. Fire-and-forget через SMTP
- Лендинг `/for-companies` (бесплатность в hero, FAQ, контакт поддержки),
  ссылки в шапке и футере
- Поддержка: email из `NEXT_PUBLIC_SUPPORT_EMAIL` (фолбэк
  support@lab-career.ru), mailto в футере, лендинге и баннерах
  rejected/blocked кабинета компании
- Дизайн и фазы 2-3: `docs/company-portal-design.md`

## Кабинет компании - фаза 2

- `notification_prefs` (API есть) - задел под email-дайджест
- ОТЛОЖЕНО (нужен настроенный SMTP + cron на VPS): email-outbox с дайджестом

  Сейчас письма шлются синхронно, SMTP на проде не настроен

## Кабинет вуза (фаза 1, дизайн: docs/vuz-portal-design.md)

- Самодекларация вуза студентом: блок «Учёба» в модалке профиля кабинета
  (вуз из справочника + курс + год выпуска), сразу засчитывается в агрегаты.
  Никакой верификации по почте/кодам - осознанное упрощение
- Справочник ~100 вузов (сид), поиск `/api/universities?q=` для селекта
- Публичная витрина `/universities/[slug]` + каталог `/universities`:
  публикуются при заполненном описании ЦКС, в sitemap (секция universities).
  Цель - замена мёртвого раздела «Трудоустройство» сайта вуза + бэклинк
- Кабинет ЦКС `/vuz`: дашборд агрегатов (студенты, динамика 30д, воронка
  до invited, по курсам, неактивные), команда owner/staff с инвайтами,
  редактирование витрины. Приватность: вуз видит ТОЛЬКО обезличенные
  счётчики, срезы < K=5 студентов не раскрываются (плашка cold-start)
- Инфографика (`components/vuz/VuzCharts.tsx`, в основном чистый SVG):
  KPI с count-up анимацией (`CountUp`, единственный клиентский островок),
  тренд по неделям, воронка, донат по курсам. Общий `VuzDashboardView`
  для живого `/vuz` и демо
- Бенчмарк (продающая метрика §4.1): «студенты вуза активнее среднего по
  платформе на N%» - обезличенное сравнение (отклики на студента, доля с
  AI-разбором) со средним по всем задекларировавшим вуз. Показывается
  только выше порогов (вуз >= K, платформа >= 20 студентов)
- Онбординг вуза ручной: `/admin/universities` (создать + инвайт owner ЦКС);
  самозаписи вуза нет, поэтому нет и премодерации
- ОТЛОЖЕНО (фазы 2-3): рассылка студентам от ЦКС, выгрузки-отчёты,
  переключатель активного вуза, «подтверждённая» доля по домену почты,
  UTM-предзаполнение вуза при регистрации

## Юзкейсы

Персоны: (1) студент/выпускник без опыта ищет стажировку, (2) джун готовится
к откликам и собеседованиям, (3) посетитель из поисковика (главный канал
трафика, цель - 70k визитов/мес), (4) зарегистрированный пользователь,
возвращается в кабинет, (5) владелец - контент-операции, (6) поисковые
боты и LLM-краулеры (индексация), (7) компания/рекрутёр - нанимает джунов
через B2B-кабинет, (8) админ платформы - модерирует компании и вакансии,
(9) сотрудник карьерного центра вуза (ЦКС) - смотрит агрегаты по своим
студентам и ведёт витрину.

30 сценариев. Ревью против кода: USECASE_REVIEW.md.

### 1. Найти стажировку и откликнуться
`/vacancies` -> фильтры (сфера/опыт/формат/тип) или поиск -> карточка ->
страница вакансии -> отклик. Два режима (`apply_mode`): external - кнопка
ведёт на внешний `apply_url` (hh.ru и т.п., все парсерные вакансии);
internal - форма отклика на платформе (`/vacancies/[slug]/apply`, требует
логина): резюме + сопроводительное, статус виден в `/office/applications`.
Сохранение вакансии (сердечко) требует логина.
Критично: пустая выдача при комбинации фильтров должна показывать
понятное "ничего не найдено", а не пустой экран; `apply_url` не должен
вести на 404 у партнёра без обработки; удалённая вакансия не должна
оставлять "битую" карточку в каталоге.

### 2. Оценить и улучшить резюме
`/tools/resume-analyzer` -> загрузка PDF/DOCX или вставка текста ->
YandexGPT -> балл + список рекомендаций -> ссылки на статьи кластера
"Резюме". Залогиненному история анализов и балл сохраняются в кабинет
(`user_resume_analyses`). Это главный магнит для регистрации.
Критично: битый/защищённый паролем PDF не должен ронять запрос без
сообщения пользователю; таймаут YandexGPT - явная ошибка, а не вечный
спиннер; гостю (без логина) результат не должен теряться при переходе
на регистрацию в том же сеансе.

### 3. Онбординг нового пользователя (чек-лист кабинета)
Регистрация -> `/office` -> чек-лист ведёт по воронке продукта
(7 пунктов, `components/office/OfficeDashboard.tsx`):
1. Резюме загружено и оценено (анализатор прямо в кабинете)
2. Заполнен профиль: направление и уровень
3. Прочитан гайд "Как написать резюме" (-> хаб resume)
4. Пройден тест на знание стека (-> хаб interview)
5. Добавлен GitHub/портфолио (-> хаб resume)
6. Сохранено 5 подходящих вакансий (-> каталог)
7. Изучен зарплатный рынок (-> калькулятор)
Профиль открывает блок "зарплата по профилю".
Критично: прогресс не должен сбрасываться при повторном входе;
пункт чек-листа, отмеченный на одном устройстве, обязан быть отмечен
и на другом (источник истины - БД, не localStorage).
ИСПРАВЛЕНО (B-1): дашборд читает `/api/checklist` при маунте и пишет
при каждой отметке; прогресс переживает перезагрузку и общий между
устройствами.

### 4. SEO-воронка: из поиска в продукт
Статья из Яндекс/Google -> внутренние ссылки (3-5 в каждой статье) ->
хаб кластера -> инструменты или вакансии -> регистрация. Поэтому статьи,
хабы и перелинковка - ядро роста; любые изменения роутов/слагов статей
опасны для трафика (есть legacy-редиректы в middleware).
Критично: смена слага статьи без парного 308-редиректа = потеря
индексации и трафика; хаб-страница не должна отдавать 500 при пустой
категории (0 статей); внутренние ссылки не должны вести на 404.

### 5. Понять свою цену
Калькулятор зарплат + дашборд `/research` + статьи кластера "Зарплата" ->
пользователь идёт на переговоры с цифрами.
Критично: дашборд не должен падать при отсутствии данных по узкой
комбинации направление/грейд/город - нужен пустой стейт, не ошибка.

### 6. Вернуться и продолжить
Повторный визит -> кабинет: сохранённые вакансии/статьи, балл резюме,
прогресс чек-листа. Сессия живёт 30 дней, логин не нужен каждый раз.
Критично: истечение сессии на 31-й день должно вести на `/login?next=...`
с сохранением намерения, а не на общую ошибку; сохранённые списки должны
быть идентичны тому, что видно в `/office` и на исходных страницах.

### 7. Контент-операции (владелец)
Статьи: `article_pipeline.py` в /Users/user/claude (10 статей/день по
семантическому ядру). Вакансии: `junior_jobs_parser/` + запись через
service role. Публикация подхватывается ISR за ~5 минут.
Критично: запись контента идёт только через service role (не anon-ключ) -
случайное использование anon-ключа в новом коде откроет запись всем;
неопубликованный (`is_published=false`) контент не должен попадать
в sitemap/каталог/поиск по сайту.
С появлением B2B инвариант уточнён: web/ теперь тоже пишет в `vacancies`
(через service role в `lib/company/vacancies.ts`), но исключительно
через статусную машину с премодерацией (см. UC-21) - прямых записей
в обход статусов быть не должно.

### 8. Регистрация и вход
`/login` -> email+пароль или "Войти через Яндекс". Email: форма ->
`/api/auth/register` или `/api/auth/login` -> session-cookie на 30 дней ->
редирект на `next` (по умолчанию `/office`). Яндекс: `/api/auth/yandex` ->
OAuth -> `/auth/callback` -> та же cookie.
Критично: отказ пользователя дать доступ Яндексу должен вернуть на
`/login` с понятной ошибкой, не крашить callback; повторный email при
регистрации - явная ошибка об уникальности (`careerlab_accounts`);
просроченная/невалидная cookie не должна пускать в `/office*`.

### 9. Восстановление пароля
`/forgot-password` -> `/api/auth/forgot-password` создаёт токен в
`careerlab_password_resets`, шлёт письмо по SMTP -> `/reset-password?token=`
-> `/api/auth/reset-password`.
Критично: токен одноразовый и с истечением - повторное использование
отклоняется; сбой SMTP не должен отдавать пользователю "письмо отправлено";
невалидный/просроченный токен на `/reset-password` - явная ошибка с
подсказкой запросить новую ссылку, не тихий редирект.

### 10. Демо-кабинет как воронка регистрации
`/office-demo` (noindex, без входа) показывает UI кабинета на заглушках ->
CTA на `/login`. Ничего не пишет в реальные пользовательские таблицы.
Критично: демо не должно случайно читать/писать данные настоящего
залогиненного пользователя; `noindex` не должен потеряться при правках
страницы (иначе демо-заглушки попадут в поиск).

### 11. Управление сохранёнками
Сердечко на карточке вакансии/статьи (каталог, страница вакансии, кабинет)
-> `/api/saved-vacancies` или `/api/saved-articles` (POST добавить,
DELETE убрать), список синхронизируется в `/office` и
`/office/saved-vacancies`.
Критично: вакансия, снятая с публикации после сохранения, не должна
ломать список сохранённых (обработать orphan slug); повторный клик не
должен плодить дубли; запрос без сессии - 401, а не тихий no-op.

### 12. Согласие на обработку персональных данных (152-ФЗ)
`/consent` + чекбокс согласия в формах (регистрация, гостевой анализатор
резюме). Факт согласия сохраняется.
Критично: форму нельзя отправить без отметки согласия; текст на
`/consent` и `/privacy-policy` должен оставаться согласован при правках
любой из двух страниц.

### 13. Индексация роботами (поисковики и LLM-краулеры)
`/sitemap.xml` -> секции (`/sitemap/[...slug]`) -> `/llms.txt` (список
статей, revalidate 1ч).
Критично: новая статья/вакансия обязана появиться в sitemap без ручных
действий; `llms.txt` не должен отставать больше заявленного часа кеша;
секция sitemap для пустой категории - валидный XML, не 500.
Осознанное ограничение (R-3): `llms.txt` отдаёт последние 1000 статей
(`LLMS_ARTICLE_LIMIT` в `app/llms.txt/route.ts`), сортировка по
`published_at desc` - это ротация, а не полный список. Свежие статьи всегда
в фиде, самые старые за порогом 1000 выпадают. Порог поднят с 200; при
приближении каталога к 1000 статей - поднять лимит в роуте и здесь.

### 14. Миграция старых ссылок
Внешние ссылки на legacy `.html`-роуты (соцсети, старые публикации) ->
middleware 308 на новые маршруты (`/vacancy.html?slug=X` -> `/vacancies/X`,
`/kb-article.html?id=X` -> `/knowledge-base/X`).
Критично: редирект обязан сохранять slug/id 1-в-1; при смене схемы
слагов статей нужен парный редирект (прецедент: `perehodit` ->
`perekhodit`), иначе теряется уже проиндексированный трафик.

### 15. Защита анализатора резюме от злоупотреблений
Гость и залогиненный дергают `/api/analyze-resume` -> rate-limit
(`lib/rate-limit.ts`) до вызова YandexGPT.
Критично: лимит должен считаться по IP/сессии, а не глобально (иначе
пиковая нагрузка одного пользователя блокирует всех); превышение лимита -
понятная ошибка, не общий 500; таймаут/сбой самого YandexGPT не должен
списываться как использованная попытка лимита.
ИСПРАВЛЕНО (B-6): rate-limit по IP подключён к analyze-resume (10/час),
login (10/15мин), register (5/час); apply и инвайты уже использовали его.

### 16. Просмотр архивной вакансии (SEO-хвост)
Вакансия закрыта -> `is_archived=true` -> из каталога исчезает
(фильтр в `listVacancies`), но страница `/vacancies/[slug]` продолжает
отдаваться с пометкой `[Архив]` в title и блоком похожих активных
вакансий - трафик из поиска конвертируется в живые вакансии.
Критично: архивная вакансия не должна появляться в каталоге, подборках
и "похожих"; страница обязана явно показывать, что вакансия закрыта,
и предлагать альтернативы, а не выглядеть живой.

### 17. Подписка на Telegram-канал (retention-канал)
Липкая плашка снизу (`components/TgStickyBar.tsx`) появляется по скроллу,
ведёт в Telegram-каналы по направлениям. Закрытие запоминается в
localStorage (`tg_bar_dismissed_until`) на время, не навсегда.
Критично: плашка не должна перекрывать контент и CTA на мобильных;
закрытие обязано уважаться до истечения срока; ссылки-инвайты должны
оставаться живыми (протухший инвайт = молчаливая потеря канала retention).

### 18. Cookie-согласие и аналитика
Первый визит -> баннер cookie (`components/CookieBanner.tsx`) ->
"Принять"/"Отклонить" пишется в localStorage (`cookie_consent`).
Критично: выбор "Отклонить" обязан реально влиять на загрузку
аналитики (Яндекс.Метрика в `app/layout.tsx`) - иначе баннер
декоративен и создаёт ложное ощущение соответствия требованиям.
ИСПРАВЛЕНО (B-8): Метрика (`components/YandexMetrika.tsx`) грузится
только при согласии "Принять" и включается сразу по событию от баннера.

### 19. Регистрация компании и верификация (B2B, анти-скам)
`/for-companies` (лендинг) -> `/company/new` (нужен логин) ->
`POST /api/companies` создаёт компанию со status=pending и owner-членство
-> письмо админам -> админ на `/admin/moderation` подтверждает
(`verified`, опционально `trusted`) или отклоняет с причиной.
До верификации компания не может отправить вакансию на публикацию.
Критично: неверифицированная компания не должна публиковаться ни при
каких переходах статусов; отклонение обязано доносить причину владельцу;
если PLATFORM_ADMIN_EMAILS пуст - компании зависнут в pending навсегда,
это должно быть видно владельцу платформы (мониторинг очереди).

### 20. Команда компании: приглашения и роли
Owner (или admin) -> `/company/team` -> инвайт по email
(`POST /api/company/[id]/invites`, токен хеширован, 7 дней, письмо со
ссылкой `/company-invite?token=`) -> приглашённый логинится под тем же
email и принимает. Роли: owner > admin > recruiter; admin управляет
командой и профилем, но не трогает владельцев; нельзя деактивировать
последнего owner.
Критично: инвайт обязан быть одноразовым и с истечением; email сессии
должен совпадать с email инвайта; понижение/деактивация не должны
оставлять компанию без owner; при недоставленном письме владелец должен
понимать, что приглашение не дошло (не молчаливый "ok").

### 21. Вакансия компании: статусная машина и премодерация
`/company/vacancies/new` -> draft -> submit -> pending_review ->
админ approve/reject -> published/rejected; published -> archive ->
unarchive (снова через модерацию). Для trusted-компаний submit публикует
сразу. Правка published-вакансии не-trusted компанией возвращает её
на модерацию. Записи - через service role, статусная машина держит
`is_published`/`is_archived` согласованными.
Критично: draft/pending/rejected никогда не видны в каталоге и sitemap;
submit требует company.status=verified; правка published не должна
"оставлять" старую версию видимой, минуя модерацию; slug вакансии
компании не должен коллизить с парсерными.

### 22. Внутренний отклик кандидата и его судьба
Вакансия с apply_mode=internal -> форма на `/vacancies/[slug]/apply`
(логин обязателен): резюме PDF/DOCX до 5МБ + сопроводительное + контакт
-> лимит 10 откликов/сутки -> команде письмо + in-app -> кандидат видит
статусы в `/office/applications` и может отозвать отклик (файл резюме
удаляется).
Критично: повторный отклик на ту же вакансию - явный 409, не дубль;
файл резюме доступен только кандидату, членам компании и админу
(никакого публичного URL); отзыв отклика обязан удалять файл; вакансия
не-published/архивная не должна принимать отклики.

### 23. Воронка откликов рекрутёра
`/company/applications` -> фильтры по вакансии и статусу -> отклик:
new -> viewed -> invited/rejected (+note) -> кандидату письмо и in-app
уведомление о смене статуса.
Критично: статусы виден только членам своей компании; смена статуса
обязана уведомлять кандидата; note не должен утекать кандидату, если
он внутренний.

### 24. Модерация платформы (админ)
`/admin/moderation` (доступ по PLATFORM_ADMIN_EMAILS) -> очередь:
компании pending + вакансии pending_review -> approve/reject с причиной
-> письма и in-app уведомления командам.
Критично: страница и API обязаны проверять email админа (не только
сессию); reject без причины бесполезен для компании - причина
обязательна в UX; пустой PLATFORM_ADMIN_EMAILS = никто не админ,
очередь копится молча.

### 25. In-app уведомления
Колокольчик в шапке (`NotificationBell`, поллинг 60с) ->
`/api/notifications` (401 гостю - колокольчик прячется) -> список +
счётчик непрочитанных -> отметка прочитанными. Пишутся при отклике,
смене статуса, итогах модерации - рядом с email.
Критично: уведомления не должны теряться, если email не ушёл (in-app -
резервный канал при ненастроенном SMTP); поллинг не должен долбить API
для гостя; отметка "прочитано" не должна затирать чужие уведомления.

### 26. Студент декларирует вуз (самодекларация, персоны 1/4)
`/office` -> модалка «Редактировать профиль» -> блок «Учёба»: поиск вуза
из справочника (`/api/universities?q=`, datalist), курс, год выпуска ->
`PUT /api/office/student-profile` -> строка в `student_profiles`. Никакой
верификации по почте/коду - осознанное упрощение (см. решение продукта).
С этого момента студент попадает в обезличенные агрегаты своего вуза.
Критично: курс/год выпуска должны сохраняться, даже если название вуза
набрано руками и не совпало с подсказкой (иначе часть формы молча
теряется - ИСПРАВЛЕНО V-1); пустое поле вуза = снять привязку, а не
оставить старую; сменил вуз - обнуляется флаг подтверждения домена;
скрытый (hidden) вуз не предлагается в поиске, но уже выбравшие его
студенты привязку сохраняют.

### 27. Сотрудник ЦКС смотрит дашборд (персона 9)
Приглашение -> `/vuz-invite?token=` -> членство active -> `/vuz`:
KPI (студенты + динамика 30д, отклики, сохранёнки, AI-разборы), тренд
по неделям, воронка до invited, распределение по курсам, неактивные.
Все агрегаты обезличены (`lib/university/stats.ts`).
Критично: вуз НИКОГДА не видит студентов поимённо - только счётчики;
срез меньше порога K=5 не раскрывается (в т.ч. группа по курсу); пока
студентов < K - показывается cold-start плашка с прогрессом к порогу и
ссылкой на демо, а не пустой или вводящий в заблуждение экран; дашборд
не должен падать на нулевых данных (generate_series держит ось тренда).

### 28. ЦКС ведёт публичную витрину (персона 9, SEO)
`/vuz/settings` -> описание ЦКС, контакты, логотип, тумблер публичных
цифр -> `PATCH /api/university/[id]` -> `/universities/[slug]`
публикуется (SSG/ISR ~10 мин), попадает в каталог `/universities` и
sitemap. Цель - заменить мёртвый раздел «Трудоустройство» сайта вуза и
дать обратную ссылку (SEO).
Критично: витрина публикуется только при заполненном описании (пустой
вуз из сида - 404, не позорит выдачу); публичное число студентов
показывается лишь при включённом тумблере И выше порога K (иначе не
показывается вовсе); название/город/статус вуза меняет только админ
платформы, ЦКС - лишь контентные поля витрины; контакты берутся из
белого списка ключей (email/phone/telegram/website/address), прочее
игнорируется.

### 29. Онбординг вуза админом платформы (персона 8)
`/admin/universities` -> создать вуз (name/short_name/city) ИЛИ найти
существующий -> пригласить owner ЦКС по email (`POST /api/admin/
universities/[id]/invites`, токен 7 дней). Ссылка-инвайт возвращается
всегда - без SMTP её передают вручную (часто в ТГ).
Критично: самозаписи вуза нет (в отличие от компаний), поэтому нет и
премодерации; вуз из сида уже существует - создавать одноимённый нельзя,
иначе появится дубликат со слагом `-2` (РИСК V-3, проверять перед
созданием); инвайт действует на конкретный email - принять можно только
войдя под ним; скрыть вуз (hidden) убирает его из поиска и витрины, но
не рвёт привязки студентов.

### 30. Демо-кабинет для продажи (персона 5/9)
`/vuz-demo` (noindex, без входа) -> та же инфографика (`VuzDashboardView`),
что и живой `/vuz`, но на иллюстративных данных нейтрального
«Университета-примера». Инструмент продажи: показать вузу «как будет».
Критично: цифры честно помечены «данные иллюстративные» и не привязаны
к реальному вузу (нельзя выдавать демо за настоящие данные конкретного
вуза); страница не должна писать ничего в БД; не под guard-ом (публичная),
но noindex - не должна попадать в выдачу.
