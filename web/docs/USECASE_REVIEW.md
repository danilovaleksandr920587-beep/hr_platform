## Найденные баги (подтверждены по коду)

### B-1. [ИСПРАВЛЕНО] Чек-лист кабинета не сохранялся (UC-3, критично)
`components/office/OfficeDashboard.tsx` - `checkRows` живёт в `useState`,
ни `/api/checklist`, ни localStorage не используются. Перезагрузка страницы
сбрасывает прогресс (кроме пункта 1, который выводится из балла резюме).
API `/api/checklist` и таблица `user_checklist_progress` существуют,
но дашборд к ним не подключён. Ломает онбординг-воронку: пользователь
не видит накопленного прогресса и теряет мотивацию возвращаться.
Фикс: читать прогресс при маунте, писать при отметке пункта.

### B-2. [ИСПРАВЛЕНО] Гостевой анализ резюме терялся после логина (UC-2, критично)
`lib/client/resume-analysis.ts` - результаты хранятся под ключом
`careerlab-resume-analysis:<scope>`. Гость пишет под нескоупленным ключом,
после регистрации чтение идёт по скоупленному - результат "исчезает".
В `lib/client/saved-items.ts` для сохранёнок есть миграция
`migrateLegacyToScope`, для анализов резюме аналога нет. Анализатор -
главный магнит регистрации, и ровно в момент конверсии пользователь
теряет свой результат. Фикс: миграция гостевого ключа в скоупленный
при первом чтении (по образцу saved-items).

### B-3. [ИСПРАВЛЕНО] Сохранёнки могут молча расходиться с БД (UC-6/UC-11, критично)
`lib/client/saved-items.ts` - источник истины localStorage, запись в БД
fire-and-forget с `catch(() => {})`. Если POST упал (сеть, истёкшая
сессия), UI показывает "сохранено", в БД записи нет. При следующем
первом визите в кабинет `syncSavedFromDb` перезаписывает localStorage
состоянием БД - несинхронизированная сохранёнка исчезает без следа.
Фикс: `setVacancySaved`/`setArticleSaved` для залогиненного (scope
непустой) ждут ответ БД и при неуспехе откатывают localStorage и сердечко
в исходное состояние (только если пользователь не переключил его заново),
плюс шлют `SAVED_ITEMS_ERROR` -> `SavedItemsToast` показывает, что не
сохранилось. Для гостя (scope пуст) поведение прежнее: localStorage без
отката (сохранёнки живут только локально - это ок). Худший сценарий потери
уже был закрыт B-7 (GET отдаёт 401/503, а не пустой список).

### B-4. [ИСПРАВЛЕНО] OAuth-вход через Яндекс без защиты от login CSRF (UC-8, критично, security)
`app/api/auth/yandex/route.ts` + `app/auth/callback/route.ts` - `state`
кодирует только `{p:"yandex", next}`, без случайного nonce, и никакой
cookie для сверки не выставляется на шаге редиректа в Яндекс. Callback
слепо доверяет пришедшему `code`/`state` и логинит браузер в тот
аккаунт, чей `code` был предъявлен - классический login CSRF: атакующий
проходит OAuth под своим аккаунтом, перехватывает свою же ссылку
`/auth/callback?code=...&state=...` (не даёт браузеру её открыть) и
подсовывает жертве. Жертва открывает ссылку - и её браузер получает
session-cookie аккаунта атакующего. Дальше жертва, думая что работает
в своём кабинете, может сохранить резюме/данные - которые атакующий
потом увидит, зайдя в "свой" аккаунт.
Фикс: генерировать случайный nonce, класть в короткоживущую httpOnly
cookie на шаге `/api/auth/yandex`, сверять с `state` в callback.

### B-5. [ИСПРАВЛЕНО] Согласие на обработку ПД (152-ФЗ) не проверяется и не хранится на сервере (UC-12, критично, compliance)
Два независимых пробела:
1. Гостевой анализатор резюме (`components/office/InlineResumeAnalyzer.tsx`,
   переиспользуется в `/tools/resume-analyzer`) вообще не показывает
   чекбокс согласия - хотя FEATURES.md прямо заявляет, что он есть.
   Резюме с ПД гостя уходит в YandexGPT без какого-либо согласия в UI.
2. Даже там, где чекбокс есть (`LoginForm.tsx`, регистрация), проверка
   `if (!consent) return` - только на клиенте. `app/api/auth/register/route.ts`
   не принимает и не сохраняет поле согласия вообще - вызов API напрямую
   (curl/fetch, минуя форму) создаёт аккаунт без какой-либо отметки о
   согласии, и даже через форму факт согласия нигде не остаётся для
   последующего аудита (в `careerlab_accounts` такого поля нет).
Фикс: чекбокс в анализаторе (`InlineResumeAnalyzer`, блокирует кнопку анализа
до отметки, ссылка на `/consent`) + `register/route.ts` проверяет
`consent === true` (иначе 400) и пишет `consent_at`/`consent_version`
(`lib/legal/consent.ts`, `CONSENT_VERSION`) в `careerlab_accounts`. Новые
колонки - миграция `20260707000000_careerlab_accounts_consent.sql`, на проде
применяется РУКАМИ ДО деплоя (user-таблица, миграций для неё нет).

### B-6. [ИСПРАВЛЕНО] Rate-limit не подключён к дорогим/чувствительным эндпоинтам (UC-15, критично, cost/security)
`lib/rate-limit.ts` существует; после B2B-мержа его используют
`/api/vacancies/[slug]/apply` (10/сутки) и инвайты - но по-прежнему
НЕ защищены: `/api/analyze-resume` (каждый вызов - платный запрос в
YandexGPT, злоумышленник может сжигать бюджет циклом), `/api/auth/login`

У `/api/auth/forgot-password` свой кулдаун 60с через БД - он работает.
Фикс: подключить rateLimit по IP к analyze-resume, login, register.
(Дока API.md исправлена в этом же ревью.)

### B-7. [ИСПРАВЛЕНО] Сбой БД стирал сохранёнки пользователя (UC-6/UC-11, критично, усиливает B-3)
`app/api/saved-vacancies/route.ts` и `saved-articles` - GET при ошибке
БД возвращает `200 {slugs: []}` вместо 5xx. `syncSavedFromDb` проверяет
только `res.ok`, принимает пустой список за истину и перезаписывает
localStorage пустотой. Итог: транзиентный сбой Postgres в момент захода
в кабинет = у пользователя "исчезают" все сохранёнки (localStorage-копия
затирается). Фикс: отдавать 503 при ошибке БД в GET, чтобы sync
пропускал перезапись.

### B-9. [ИСПРАВЛЕНО] Инвайты в команду молча уходили в никуда без SMTP (UC-20, критично для B2B)
`app/api/company/[id]/invites/route.ts` - токен уходит только в письмо
(`notifyCompanyInvite`, fire-and-forget), ответ API - голый `{ok: true}`

не существует нигде, кроме несуществующего письма - позвать команду
физически невозможно. Той же проблеме подвержены все B2B-письма
(новый отклик, итоги модерации), но у них есть резерв - in-app
уведомления; у инвайта резерва нет.
Фикс-минимум: возвращать inviteUrl в ответе API и показывать owner-у
"скопируйте ссылку"; фикс-максимум - настроить SMTP.

### B-10. [ИСПРАВЛЕНО] Суточный лимит откликов сжигался невалидными попытками (UC-22)
`app/api/vacancies/[slug]/apply/route.ts` - `rateLimit()` вызывается

отклика. Вдобавок лимит in-memory (R-1): рестарт pm2 его обнуляет.
Фикс: списывать лимит после валидации (перед createApplication) или
возвращать попытку при ошибке валидации.

### B-8. [ИСПРАВЛЕНО] Cookie-баннер декоративен: "Отклонить" ни на что не влияет (UC-18, compliance)

"Отклонить", а метрика с cookie ставится всё равно. Баннер создаёт
ложное ощущение соответствия требованиям.
Фикс: грузить Метрику только после `accepted` (и по событию из баннера).

## Риски (не баг сейчас, сломается при изменении условий)

### R-1. [ИЗВЕСТНОЕ ОГРАНИЧЕНИЕ] Rate-limit - in-memory per-process (UC-15, вдобавок к B-6)
Даже когда `lib/rate-limit.ts` подключат (B-6), сама реализация - Map
в памяти процесса. При рестарте pm2 лимиты обнуляются; при нескольких
инстансах каждый считает свой лимит независимо. Комментарий в коде сам
предлагает Redis для multi-instance.
Статус: осознанно НЕ исправляем. Актуально только при мультиинстансе; сейчас
один инстанс pm2, поэтому лимиты консистентны. Не вводим Redis без явной
потребности (лишняя инфра-зависимость). Пересмотреть при переходе на несколько
инстансов / горизонтальном масштабировании - тогда вынести счётчики в Redis.

### R-2. [ИСПРАВЛЕНО] Расхождение документации и кода (UC-3)
FEATURES.md описывал 5 пунктов чек-листа, в коде их 7. Конкретное расхождение
исправлено в этом же ревью. Как процессная защита от повторения: правило проекта
«тронул страницу/роут/таблицу - обнови docs/ в том же коммите» + автопроверка
`npm run docs:check` (сверяет docs/ и app/). Остаётся дисциплиной ревьюера для
случаев, которые скрипт не ловит (содержимое онбординга, тексты фич).

### R-4. [ИСПРАВЛЕНО] Данные калькулятора зарплат и /research захардкожены без отметки актуальности (UC-5)
`components/tools/SalaryCalculatorPage.tsx` и
`components/research/researchSalaryData.ts` - все цифры зашиты в код.
Пустых стейтов не бывает (плюс), но данные молча устареют: нет ни даты
актуальности в UI, ни процесса обновления. Пользователь пойдёт на
переговоры с цифрами 2025 года в 2027-м.
Фикс:
- Калькулятор зарплат: данные индексируются от `SALARY_AS_OF`
  (`lib/data/salary.ts`), в UI выводится бейдж «данные на <месяц год>»
  через `salaryAsOfLabel()`.
- `/research`: цифры - статический снимок, поэтому добавлена фиксированная
  константа `RESEARCH_AS_OF` (`researchSalaryData.ts`) и видимая пометка
  «Данные на <дата>. Обновляем раз в квартал» в шапке дашборда.
- Процесс: обновлять раз в квартал - при обновлении править числа И дату
  (`SALARY_AS_OF` / `RESEARCH_AS_OF`).

### R-5. [ИСПРАВЛЕНО] Ссылки в письмах строились из req.url (UC-9, UC-20)
`app/api/auth/forgot-password/route.ts` и
`app/api/company/[id]/invites/route.ts` - `new URL(req.url).origin`.
За nginx-прокси origin зависит от заголовка Host и схемы, которую видит
Next (обычно http). В письме может уйти `http://` вместо `https://` или
внутренний хост - в зависимости от конфига прокси. В `auth/callback`
эта задача уже решена через `getRealOrigin()` (x-forwarded-proto/host) -
здесь стоит использовать её же.

### R-3. [ЗАДОКУМЕНТИРОВАНО] llms.txt обрезается на 200 самых новых статей (UC-13)
`app/llms.txt/route.ts` - `listArticles({ limit: 200 })`, сортировка по
`published_at desc`. Новые статьи всегда попадают в фид (это ок), но
после 200-й опубликованной статьи самые старые начинают молча выпадать
из `llms.txt` - LLM-краулер перестаёт их видеть без какого-либо сигнала
об этом. При темпе публикаций 10 статей/день (см. FEATURES.md) порог
в 200 статей достигается быстро. Лимит нигде не задокументирован как
осознанное ограничение.
Фикс: лимит поднят 200 -> 1000 (`LLMS_ARTICLE_LIMIT` в `app/llms.txt/route.ts`)
и задокументирован как осознанная ротация - в комментарии роута и в FEATURES.md
(§13). Полностью снять лимит нельзя надёжно: PostgREST режет ответ по db-max-rows,
т.е. «безлимит» дал бы такой же обрез, только неявный.

- Файл резюме отклика: доступ только кандидату/членам компании/админу,
  `path.basename` против traversal, вайтлист расширений, 5 МБ,
  `Cache-Control: private, no-store` (UC-22)
- Инвайты: токен хеширован, одноразовый, 7 дней, email сессии обязан
  совпасть с email приглашения (UC-20)
- Статусная машина вакансий держит `is_published`/`is_archived`
  согласованно: draft/pending/rejected невидимы в каталоге и sitemap;
  правка published не-trusted компанией возвращает на модерацию (UC-21)
- `/admin/moderation`: и страница, и API проверяют PLATFORM_ADMIN_EMAILS,
  не только сессию (UC-24)
- Повторный отклик гасится и проверкой, и unique-констрейнтом с откатом
  файла резюме; отзыв отклика удаляет файл (UC-22)

## Возможности для развития (пробелы в сценариях)

1. **Retention-петля отсутствует.** UC-6 "вернуться и продолжить"
   полагается на то, что пользователь сам вспомнит про сайт. Нет ни
   email-дайджеста новых вакансий по профилю, ни уведомлений. Профиль
   (направление/уровень/формат/город) уже собирается - подборка под него
   уже строится (`matchedVacancies` в кабинете), остался канал доставки.
2. **Воронка отклика обрывается на apply_url.** Платформа не знает,
   откликнулся ли пользователь. Дешёвый шаг: статус "откликнулся" на
   сохранённой вакансии - и кабинет превращается в трекер откликов,
   самую липкую функцию для активного соискателя.
3. **История баллов резюме как график прогресса.** Данные уже копятся
   в `user_resume_analyses` (score, дата). График "балл растёт" -
   наглядная ценность кабинета и повод перепроверять резюме после правок.
4. **Пустые стейты как класс задач.** Несколько критичных точек юзкейсов -
   про деградацию UX (пустая выдача фильтров, пустой хаб, нулевые данные
   калькулятора). Стоит пройти одним заходом и добавить осмысленные
   пустые состояния с CTA вместо пустых экранов.
5. **Чек-лист как продукт, а не виджет.** После фикса B-1 чек-лист может
   стать центром онбординга: прогресс в шапке кабинета, письмо-напоминание
   о незавершённых шагах, "следующий шаг" на главной для залогиненных.

## Контур вузов (фаза 1, UC-26..30)

Проверка сценариев кабинета вуза против кода. Найдено при ревью после
реализации; критичные исправлены в этой же ветке.

### V-1. [ИСПРАВЛЕНО] Курс/год выпуска молча терялись (UC-26, критично)
В `OfficeDashboard.saveProfile` весь `PUT /api/office/student-profile`
вызывался только если название вуза пустое, совпало с подсказкой или
равно сохранённому. Если студент печатал вуз руками (без выбора из
datalist) и при этом менял курс/год - запрос не уходил вовсе, изменения
курса/года пропадали без ошибки. Фикс: курс и год пишутся всегда; вуз -
отдельной логикой (пусто -> null/очистить, совпал -> id/сменить, набран
вручную -> ключ не отправляем = не менять привязку).

### V-2. [ИСПРАВЛЕНО] Донат вырождался на одном сегменте (UC-27, визуальное)
`DonutChart` рисовался дугами через arcPath; сегмент на 100% (например
все студенты одного курса) давал start==end и невидимую дугу. Переписан
на кольцо со `stroke-dasharray`: корректно рисует и один сегмент, и много;
удалены ненужные хелперы polar/arcPath.

### V-3. [РИСК] Дубликат вуза при онбординге (UC-29)
`/admin/universities` -> «Создать вуз» с названием, которое уже есть в
сиде, сгенерирует новый слаг `-2` (uniqueUniversitySlug) - в справочнике
окажутся два МИСИС, студенты разойдутся по обоим, агрегаты разъедутся.
Сейчас защищает только внимательность оператора (список вузов на той же
странице). Развитие: при создании подсвечивать похожие по названию и
предлагать открыть существующий вместо создания.

### V-4. [ПО ЗАДУМКЕ] Витрина публикуется с лагом ISR (UC-28)
Правка описания в `/vuz/settings` попадает на публичную витрину и в
каталог не мгновенно, а в пределах revalidate (600с). Это осознанный
кэш; форма прямо предупреждает «обновится в течение ~10 минут». Багом
не является, но при демо клиенту стоит учитывать.

### V-5. [ПО ЗАДУМКЕ] Верификации студента нет (UC-26)
Привязка студент->вуз - чистая самодекларация, никакой проверки по
корпоративной почте или коду ЦКС. Осознанное упрощение (скорость запуска
важнее строгости на этапе пилота). Следствие: цифры дашборда отражают
«кто сам себя отнёс к вузу». Флаг `student_profiles.email_verified_domain`
зарезервирован под будущую «подтверждённую» долю (фаза 2-3).

### Проверено - ок (вузы)
- Приватность: `lib/university/stats.ts` отдаёт только счётчики; при
  `studentCount < K` весь дашборд заменяется cold-start плашкой, срез по
  курсу с группой < K не попадает в донат (UC-27)
- Публичные цифры витрины: `getPublicUniversityStats` возвращает null,
  если тумблер выключен ИЛИ студентов < K - фейковых чисел не показать (UC-28)
- Guard: и страницы `/vuz*` (getActiveUniversity), и API
  (`requireUniversityRole`) проверяют членство и роль, не только сессию;
  `/vuz-demo` намеренно публичный, но noindex (UC-27, UC-30)
- Смена роли/деактивация члена ЦКС не даёт снять роль с себя - вуз не
  останется без owner (UC-27, `members/[acc]`)
- Инвайт-токен привязан к email и сроку (7 дней); принять можно только
  войдя под этим email (UC-29, общий механизм с company_invites)
- Онбординг-инвайты админа и кабинета разведены по разным rate-limit
  бакетам (`:admin:` / `:cabinet:`), не сжигают лимит друг друга

## Возможности для развития (контур вузов)

6. **[РЕАЛИЗОВАНО]** Бенчмарк-сравнение - «студенты вуза активнее среднего
   по платформе на N%» (отклики на студента, доля с AI-разбором). Считается
   в `lib/university/stats.ts` (поле `benchmark`), рендерится `BenchmarkBars`.

   (`CountUp`). Самая продающая метрика дизайн-дока §4.1.

8. **Экспорт-отчёт для ректората** - PDF/скриншот-готовая страница
   с агрегатами и логотипом вуза (P1 из дизайн-дока §4.5).
9. **[РЕАЛИЗОВАНО]** Онбординг-чеклист ЦКС (`VuzChecklist`) - активация
   «N из 5» (витрина, контакты, публичные цифры, коллега, первые студенты).
   Состояние из реальных данных, на `/vuz`; исчезает, когда всё сделано.
10. **Анти-дубликат вуза** (см. V-3) - подсказка похожих при создании.
